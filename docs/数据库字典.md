## 数据库概述

本数据库为购物平台实训项目的核心存储系统，包含7张核心业务表，支持用户管理、商品管理、购物车、订单处理等完整电商流程。

### 核心设计思想

1. **订单分表设计**：订单分为 `orders`（订单头）和 `order_items`（订单明细）两张表，实现1:N关系的规范化设计
2. **价格快照机制**：订单明细表独立存储下单时的商品价格，不受后续商品调价影响
3. **复合唯一键**：购物车表使用 (user_id, product_id) 复合唯一键，配合 ON DUPLICATE KEY UPDATE 实现高效的增量更新
4. **软删除设计**：商品和分类使用 status 字段标记状态，避免硬删除导致历史数据关联丢失
5. **自关联设计**：分类表使用 parent_id 实现树形结构，支持多级分类

## 表结构详情

### 1. users（用户表）

**表说明**：存储平台所有用户信息，包括普通用户和管理员

| 字段名 | 数据类型 | 允许NULL | 键类型 | 默认值 | 说明 |
|--------|----------|----------|--------|--------|------|
| id | INT | NO | PRI | AUTO_INCREMENT | 用户ID（主键） |
| username | VARCHAR(50) | NO | UNI | 无 | 用户名（唯一） |
| password | VARCHAR(255) | NO | - | 无 | 密码（bcrypt哈希，以$2a$开头） |
| email | VARCHAR(100) | YES | - | NULL | 邮箱 |
| phone | VARCHAR(20) | YES | - | NULL | 手机号 |
| role | ENUM('user','admin') | NO | - | 'user' | 角色（user=普通用户，admin=管理员） |
| status | TINYINT | NO | - | 1 | 状态（1=正常，0=禁用） |
| created_at | DATETIME | NO | - | CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | NO | - | CURRENT_TIMESTAMP ON UPDATE | 更新时间 |

**索引**：
- PRIMARY KEY (`id`)
- UNIQUE KEY (`username`)

**教学重点**：
- ✅ 用户名唯一约束（UNIQUE）
- ✅ 密码字段长度设置为255（bcrypt哈希结果约60字符，预留空间）
- ✅ ENUM类型限制角色只能是特定值
- ✅ 时间戳自动更新（ON UPDATE CURRENT_TIMESTAMP）

**索引**：
- PRIMARY KEY (`id`)

- ✅ 自关联设计（parent_id引用同表的id）

### 3. products（商品表）

**表说明**：商品基本信息，包括价格、库存、分类关联等

| 字段名 | 数据类型 | 允许NULL | 键类型 | 默认值 | 说明 |
|--------|----------|----------|--------|--------|------|
| id | INT | NO | PRI | AUTO_INCREMENT | 商品ID（主键） |
| name | VARCHAR(200) | NO | - | 无 | 商品名称 |
| description | TEXT | YES | - | NULL | 商品描述 |
| price | DECIMAL(10,2) | NO | MUL | 无 | 售价（精确到分） |
| stock | INT | NO | MUL | 0 | 库存数量 |
| category_id | INT | YES | MUL | NULL | 分类ID（外键） |
| image_url | VARCHAR(300) | YES | - | NULL | 商品图片URL |
| status | TINYINT | NO | MUL | 1 | 状态（1=上架，0=下架） |
| created_at | DATETIME | NO | - | CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | NO | - | CURRENT_TIMESTAMP ON UPDATE | 更新时间 |

**索引**：
- PRIMARY KEY (`id`)
- KEY `idx_category` (`category_id`)
- KEY `idx_price` (`price`)
- KEY `idx_stock` (`stock`)
- KEY `idx_status` (`status`)

**外键**：
- FOREIGN KEY (`category_id`) REFERENCES `categories(id)` ON DELETE SET NULL

**教学重点**：
- ✅ DECIMAL(10,2)精确小数，避免浮点数计算误差
- ✅ 多个索引优化查询（价格筛选、库存排序、状态过滤）
- ✅ 外键 ON DELETE SET NULL（分类删除时商品不删除，只是分类字段置空）
- ✅ 软删除设计（status=0下架，不物理删除）

### 4. orders（订单表）

**表说明**：订单头信息，存储订单基本信息和收货地址

| 字段名 | 数据类型 | 允许NULL | 键类型 | 默认值 | 说明 |
|--------|----------|----------|--------|--------|------|
| id | INT | NO | PRI | AUTO_INCREMENT | 订单ID（主键） |
| order_no | VARCHAR(50) | NO | UNI | 无 | 订单号（唯一，业务主键） |
| user_id | INT | NO | MUL | 无 | 用户ID（外键） |
| total_amount | DECIMAL(10,2) | NO | - | 无 | 订单总金额 |
| status | ENUM | NO | MUL | 'pending' | 订单状态 |
| address | VARCHAR(500) | YES | - | NULL | 收货地址 |
| phone | VARCHAR(20) | YES | - | NULL | 收货电话 |
| created_at | DATETIME | NO | MUL | CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | NO | - | CURRENT_TIMESTAMP ON UPDATE | 更新时间 |

**status枚举值**：
- `pending` - 待支付
- `paid` - 已支付
- `shipped` - 已发货
- `completed` - 已完成
- `cancelled` - 已取消

**索引**：
- PRIMARY KEY (`id`)
- UNIQUE KEY (`order_no`)
- KEY `idx_user` (`user_id`)
- KEY `idx_status` (`status`)
- KEY `idx_created_at` (`created_at`)

**外键**：
- FOREIGN KEY (`user_id`) REFERENCES `users(id)` ON DELETE RESTRICT

**教学重点**：
- ✅ order_no作为业务主键（不依赖自增ID）
- ✅ ENUM类型管理订单状态流转
- ✅ 外键 ON DELETE RESTRICT（用户不能删除有订单的账号）
- ✅ created_at索引支持按时间范围查询

### 5. order_items（订单明细表）

**表说明**：订单商品明细，存储订单中的具体商品信息（⭐价格快照）

| 字段名 | 数据类型 | 允许NULL | 键类型 | 默认值 | 说明 |
|--------|----------|----------|--------|--------|------|
| id | INT | NO | PRI | AUTO_INCREMENT | 订单明细ID（主键） |
| order_id | INT | NO | MUL | 无 | 订单ID（外键） |
| product_id | INT | NO | MUL | 无 | 商品ID（外键） |
| quantity | INT | NO | - | 无 | 购买数量 |
| price | DECIMAL(10,2) | NO | - | 无 | **下单时的商品价格（快照）** |
| created_at | DATETIME | NO | - | CURRENT_TIMESTAMP | 创建时间 |

**索引**：
- PRIMARY KEY (`id`)
- KEY `idx_order` (`order_id`)
- KEY `idx_product` (`product_id`)

**外键**：
- FOREIGN KEY (`order_id`) REFERENCES `orders(id)` ON DELETE CASCADE
- FOREIGN KEY (`product_id`) REFERENCES `products(id)` ON DELETE RESTRICT

**教学重点**：
- ✅ **价格快照设计**：price字段存储下单时的价格，不随商品表价格变动而变化
- ✅ 外键 ON DELETE CASCADE（订单删除时明细自动删除）
- ✅ 为什么订单要分两张表？因为一个订单包含多个商品（1:N关系）

**价格快照验证示例**：
```sql
-- 1. 下单时 iPhone 价格是 7999
INSERT INTO order_items (order_id, product_id, quantity, price) 
VALUES (1, 1, 1, 7999.00);

-- 2. 商家涨价到 8999
UPDATE products SET price = 8999.00 WHERE id = 1;

-- 3. 查询订单明细，price 仍然是 7999（不受影响）
SELECT * FROM order_items WHERE order_id = 1;
-- 结果：price = 7999.00 ✓
```

### 6. cart_items（购物车表）

**表说明**：用户购物车，存储用户加购的商品

| 字段名 | 数据类型 | 允许NULL | 键类型 | 默认值 | 说明 |
|--------|----------|----------|--------|--------|------|
| id | INT | NO | PRI | AUTO_INCREMENT | 购物车ID（主键） |
| user_id | INT | NO | MUL | 无 | 用户ID（外键） |
| product_id | INT | NO | - | 无 | 商品ID（外键） |
| quantity | INT | NO | - | 1 | 数量 |
| created_at | DATETIME | NO | - | CURRENT_TIMESTAMP | 添加时间 |
| updated_at | DATETIME | NO | - | CURRENT_TIMESTAMP ON UPDATE | 更新时间 |

**索引**：
- PRIMARY KEY (`id`)
- UNIQUE KEY `uk_user_product` (`user_id`, `product_id`)
- KEY `idx_user` (`user_id`)

**外键**：
- FOREIGN KEY (`user_id`) REFERENCES `users(id)` ON DELETE CASCADE
- FOREIGN KEY (`product_id`) REFERENCES `products(id)` ON DELETE CASCADE

**教学重点**：
- ✅ **复合唯一键**：(user_id, product_id) 保证同一用户同一商品只有一条记录
- ✅ 配合 ON DUPLICATE KEY UPDATE 实现高效更新
- ✅ 外键 ON DELETE CASCADE（用户删除时购物车自动清空）

**ON DUPLICATE KEY UPDATE 示例**：
```sql
-- 用户1添加商品100，如果已存在则数量+1
INSERT INTO cart_items (user_id, product_id, quantity) 
VALUES (1, 100, 1)
ON DUPLICATE KEY UPDATE quantity = quantity + 1;
```

## ER图关系说明

### 实体关系

```
users (1) ----< (N) orders          【一个用户可以有多个订单】
users (1) ----< (N) cart_items      【一个用户可以有多个购物车记录】
users (1) ----< (N) reviews         【一个用户可以发表多个评价】

categories (1) ----< (N) products   【一个分类包含多个商品】
categories (1) ----< (N) categories 【分类自关联，支持多级分类】

products (1) ----< (N) cart_items   【一个商品可以被多个用户加购】
products (1) ----< (N) order_items  【一个商品可以出现在多个订单明细】
products (1) ----< (N) reviews      【一个商品可以有多个评价】

orders (1) ----< (N) order_items    【一个订单包含多个商品明细】
orders (1) ----< (N) reviews        【一个订单可以关联多个评价】
```

### 外键策略

| 外键关系 | ON DELETE 策略 | 说明 |
|----------|----------------|------|
| products.category_id → categories.id | SET NULL | 分类删除时商品保留，分类字段置空 |
| orders.user_id → users.id | RESTRICT | 有订单的用户不能删除 |
| order_items.order_id → orders.id | CASCADE | 订单删除时明细自动删除 |
| order_items.product_id → products.id | RESTRICT | 有订单的商品不能删除 |
| cart_items.user_id → users.id | CASCADE | 用户删除时购物车自动清空 |
| cart_items.product_id → products.id | CASCADE | 商品删除时自动从所有购物车移除 |

## 索引设计说明

### 主键索引
所有表都使用 AUTO_INCREMENT 整型主键，查询效率高

### 唯一索引
- `users.username` - 用户名唯一，登录查询
- `orders.order_no` - 订单号唯一，业务查询
- `cart_items.(user_id, product_id)` - 复合唯一，防止重复加购

### 普通索引
- **外键字段全部加索引**：提高JOIN查询性能
- **状态字段**：status字段常用于WHERE条件
- **价格和库存**：支持筛选和排序
- **时间字段**：支持按时间范围查询

## 数据库设计亮点（答辩重点）

### 1. 为什么订单要分两张表？

**问题**：如果只有一张订单表，商品信息怎么存？

**错误方案**：在订单表里加10个字段（product1, product2, ..., product10）

**正确方案**：
- `orders` 表存订单头信息（谁买的、多少钱、地址）
- `order_items` 表存订单明细（具体买了什么、多少件、单价）
- 这是典型的 **1:N 关系规范化设计**

### 2. 价格快照机制

**问题**：用户下单后商家调价，历史订单的价格是否应该跟着变？

**答案**：不应该！用户看到的价格是下单时的价格

**实现**：
- `order_items.price` 独立存储下单时的价格（快照）
- 不要直接 JOIN products 表取价格（会受调价影响）

### 3. ON DUPLICATE KEY UPDATE 的优势

**传统方案**（3条SQL，有并发问题）：
```sql
SELECT * FROM cart_items WHERE user_id=1 AND product_id=100;
-- 如果存在：
UPDATE cart_items SET quantity=quantity+1 WHERE ...;
-- 如果不存在：
INSERT INTO cart_items ...;
```

**优化方案**（1条SQL，原子操作）：
```sql
INSERT INTO cart_items (user_id, product_id, quantity) 
VALUES (1, 100, 1)
ON DUPLICATE KEY UPDATE quantity = quantity + 1;
```

**前提**：必须有 UNIQUE KEY (user_id, product_id)
